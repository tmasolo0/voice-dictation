# WORKFLOW.md — Правила командной работы

> **Назначение:** Единые правила для всех участников проекта (люди + AI-агенты).
> **Размещение:** `.context/WORKFLOW.md` + Claude APP Project Knowledge

---

## 1. Участники и роли

| Роль | Кто | Уровень |
|------|-----|---------|
| Человек | Дмитрий / Артём / Даня | Принятие решений, инициация, ревью |
| Claude Code | У каждого свой | Тактический — код, выполнение задач |
| Claude APP | У каждого свой | Стратегический — архитектура, UX, декомпозиция |

Роли людей не фиксированы — каждый может взять любую задачу.

---

## 2. Claude Code — тактический уровень

**Получает задачи от:**
- Человека напрямую
- Из pm/TASKS.md
- Через .handoff/ (от APP)

**Может самостоятельно:**
- Декомпозировать задачу на подшаги
- Выполнять задачи последовательно по приоритету
- Обновлять pm/TASKS.md, pm/CHANGELOG.md, pm/STATUS.md (факты)
- Добавлять идеи в pm/BACKLOG.md

**Стоп-условия (пишет в pm/STATUS.md → "Требует решения" → СТОП):**
- Архитектурный выбор (два+ варианта)
- Неясность в требованиях
- Конфликт с существующей логикой
- Выход за рамки текущей задачи

---

## 3. Claude APP — стратегический уровень

**Вызывается когда:**
- Начало / завершение этапа
- Декомпозиция крупной фичи
- Архитектурные развилки
- UX/UI проектирование
- Пересмотр приоритетов

**Не обязателен для:** понятных задач и мелких правок.

---

## 4. Кто формирует задачи

| Масштаб | Кто | Пример |
|---------|-----|--------|
| Этап / крупная фича | Человек + APP | «Авторизация» → APP декомпозирует |
| Средняя задача | Человек → Code | «Добавь OAuth» → Code разбивает сам |
| Мелочь / багфикс | Человек → Code | «Поправь валидацию» → Code делает |

---

## 5. Ответственность за служебные файлы

Все файлы управления проектом находятся в `pm/`.

| Файл | Claude Code | Claude APP | Человек |
|------|------------|------------|---------|
| pm/STATUS.md | ✅ факты | ✅ стратегия | Читает, решает |
| pm/TASKS.md | ✅ статусы | ✅ формирует задачи | Ставит, ревьюит |
| pm/CHANGELOG.md | ✅ пишет | — | Читает |
| pm/ROADMAP.md | — | ✅ пишет | Утверждает |
| pm/BACKLOG.md | ✅ добавляет | ✅ добавляет | Приоритизирует |
| docs/ARCHITECTURE.md | Читает, следует | ✅ пишет | Утверждает |

---

## 6. Маркировка участников

Все изменения помечаются `@Имя`: `@Дима`, `@Артём`, `@Даня`.

**Где прописать:**
- Claude Code: `~/.claude/CLAUDE.md` → секция «Идентификация» (глобально, один раз)
- Claude APP: Project Instructions → секция «Идентификация» (при создании проекта)

**Примеры:**
- pm/TASKS.md: `✅ @Дима, 2026-02-25`
- pm/CHANGELOG.md: `@Дима — добавил JWT-авторизацию`
- pm/STATUS.md: `> Обновлено: 2026-02-25, @Дима`

## 7. Механизм .handoff/ (APP → Code)

Папка `.handoff/` в корне проекта, в `.gitignore`.

**Когда:** APP сформулировал сложную задачу с полным контекстом.

**Формат** — `.handoff/TASK_N.N.md`:

```markdown
# TASK-N.N: [Название]

## Цель
[Что нужно получить]

## Контекст
[Что уже сделано, от чего отталкиваемся]

## Требования
[Конкретные пункты]

## Файлы
[Какие файлы создать/изменить]

## Критерии готовности
[Когда задача считается выполненной]
```

**Команда:** `Выполни задачу из .handoff/TASK_N.N.md`

**Результат:** Code создаёт `.handoff/TASK_N.N_DONE.md` с отчётом.

**Очистка:** файлы удаляются после приёмки.

## 8. Форматы служебных файлов

Все файлы в `pm/`. Полные шаблоны — в заготовках соответствующих файлов.

### pm/STATUS.md
```
> Обновлено: [дата], @[Имя] ([Code/APP])
Сейчас → Последние изменения → Следующая задача → Требует решения → Блокеры
```

### pm/TASKS.md
```
TASK-{этап}.{номер}: [Название]
Статусы: 🔲 backlog → 👤 взято @Имя → 🔧 в работе @Имя → ✅ @Имя, дата
```

### pm/ROADMAP.md
```
Этапы с целями, без жёстких дат. Обновляет APP, утверждает команда.
```

### pm/CHANGELOG.md
```
[дата] — @Имя — [что сделано]
```

### pm/BACKLOG.md
```
P1 (важно) → P2 (желательно) → P3 (идеи)
```
